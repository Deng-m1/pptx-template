# `pptx-template` 切片还原工作流

本文档用于规范 `pptx-template` 项目中的切片还原流程。目标不是生成一张“数值上看起来接近”的截图，而是通过**人工分析原图、手工编写 HTML、截图后再次人工复核**的方式，把每一页页面尽量还原到与原图一致的视觉效果。

> 本项目的核心原则是：**可以使用工具完成截图，但不能使用脚本做图像相似度对比；页面是否达标，必须由人工查看原图与截图后再判断。**

## 1. 工作流总原则

切片优化必须遵循“**看原图 → 写页面 → 截图 → 人工复看 → 再修改**”的闭环。判断页面是否接近原图时，重点看的是版式结构、色块比例、文字层级、留白、装饰元素、图表形态、图例位置、边框样式、图像素材以及整体气质，而不是任何自动化分数。

自动化工具在本项目中的作用只限于**渲染与截图**，用于帮助产出当前 HTML 的可视结果，不用于代替人工审美判断。任何基于 SSIM、像素差、MSE、感知哈希或其他机器打分方式的对比，在本项目中都**不作为工作流的一部分**，也**不允许作为验收依据**。

## 2. 正确流程概览

下表给出本项目允许采用的标准流程，以及每一步的目标与输出。

| 阶段 | 操作内容 | 允许使用的方式 | 输出结果 |
| :--- | :--- | :--- | :--- |
| 原图分析 | 打开原始 PNG，人工识别布局、层级、配色、图表与素材 | 人工查看图片 | 页面结构分析结论 |
| HTML 编写 | 手工修改或重写对应切片 HTML | 文本编辑 | 更新后的 `html/slide_xxx.html` |
| 页面截图 | 渲染 HTML 并生成当前页面截图 | 截图脚本或浏览器截图 | 更新后的 `screenshots/slide_xxx.png` |
| 人工复核 | 同时查看原图与截图，逐项比较差异 | 人工查看图片 | 差异记录与下一轮修改方向 |
| 记录沉淀 | 将观察到的问题与结论写入记录文件 | 文本编辑 | `records/` 下的分析记录 |

这个流程必须循环执行，直到页面在人工观察下已经基本贴近原图为止。

## 3. 明确禁止事项

为了避免流程跑偏，以下做法在本项目中视为**错误工作流**。

| 禁止事项 | 说明 |
| :--- | :--- |
| 使用图像对比脚本判断页面优劣 | 不允许使用 SSIM、像素差、MSE、感知哈希等方式评估切片质量 |
| 依据机器分数决定是否完成优化 | 页面是否合格只能由人工看图判断，不能看脚本分数 |
| 用自动对比结果替代原图分析 | 机器只能辅助截图，不能代替人眼识别版式和设计风格 |
| 只改局部数值而不复看整页 | 每次修改后都必须重新截图并人工查看整页效果 |
| 只看 HTML 不看截图 | 最终结果必须落回图片层面对照原图检查 |

> 如果某一页“结构看起来差不多”，但字体大小、图表形态、装饰条位置、色块厚度、边距或留白不对，依然不能算完成。

## 4. 原图分析要求

在开始修改任何一个切片之前，必须先打开原图进行人工分析。分析时不要只看文字内容，而要先判断整页的**版式骨架**：是双栏页、三栏页、摘要卡片页、图表页、目录页，还是图文混排页。之后再继续拆解标题区、主视觉区、信息卡、图表、图例、说明文案、底栏、背景装饰与图片素材。

分析时建议至少回答以下问题，并将结论写入记录文件：

| 分析维度 | 应关注的问题 |
| :--- | :--- |
| 结构 | 页面是几栏结构，模块之间如何对齐，主次关系是什么 |
| 标题 | 标题字号、字重、位置、上下边距和装饰线如何安排 |
| 配色 | 主色、辅助色、浅色背景、边框色、文字色分别是什么 |
| 图表 | 是柱图、折线图、饼图、环图、地图还是装饰图形，比例是否夸张 |
| 文案 | 段落宽度、行高、字距、粗细和分段方式是否明显 |
| 细节 | 虚线、圆角、阴影、分隔线、图标、编号、角标是否存在 |

原图分析写得越清楚，后续修改越快。对于差异明显的页，通常需要先记下“最大偏差点”，例如“右侧信息面板底色错误”“标题条厚度不足”“图例位置偏左”“柱图数据关系与原图相反”等。

## 5. HTML 编写要求

HTML 还原应优先追求**视觉一致**，而不是代码形式的一致。只要最终截图更接近原图，就可以重构原有 HTML 结构。对于商务模板类切片，通常建议使用绝对定位与明确尺寸控制，以便稳定还原 `1280×720` 画布中的位置关系。

编写时应重点控制以下内容。

| 项目 | 要求 |
| :--- | :--- |
| 画布尺寸 | 页面固定为 `1280×720` |
| 布局方式 | 优先使用明确的绝对定位、固定宽高与稳定间距 |
| 文字层级 | 明确区分主标题、副标题、说明文字、数据数字和图例 |
| 色块结构 | 标题条、卡片底、侧栏、底栏、分区块要与原图厚度一致 |
| 图表造型 | 允许手写 SVG，以更准确还原原图中的非标准图表形态 |
| 素材处理 | 需要时可使用本地素材或内嵌图片，但必须保证截图稳定 |

当现有 HTML 的结构已经明显偏离原图时，应直接重构，而不是在错误骨架上做小修小补。

## 6. 截图要求

本项目允许使用截图工具来渲染当前 HTML 页面，因为截图本身属于产出环节，而不是判断环节。截图的目的，是把当前 HTML 真实呈现为图片，供人工再次观察。

推荐使用项目内已有截图脚本：

```bash
python3.11 scripts/screenshot_html.py \
  "01高级商务蓝配色/html/slide_1.html" \
  "01高级商务蓝配色/screenshots/slide_1.png"
```

如需批量更新截图，也可以使用浏览器或批量命令完成渲染；但无论采取何种截图方式，都必须满足下表要求。

| 检查项 | 要求 |
| :--- | :--- |
| 视口尺寸 | 必须与项目标准画布一致，即 `1280×720` |
| 截图对象 | 必须是当前正在优化的 HTML 页面 |
| 输出位置 | 默认输出到 `01高级商务蓝配色/screenshots/` |
| 后续动作 | 截图完成后必须立刻人工查看，不得直接结束 |

## 7. 人工复核要求

截图生成后，必须同时查看**原图**和**当前截图**。复核时应从整体到局部进行，而不是只盯某个文字或某个图表。建议先看三件事：整体构图是否一致、视觉重心是否一致、页面气质是否一致。只有这三点接近，才进入局部校正。

人工复核时建议重点检查以下方面。

| 复核维度 | 典型问题 |
| :--- | :--- |
| 整体比例 | 左右栏宽度不对、卡片过高或过矮、模块留白失衡 |
| 标题系统 | 标题偏小、太粗、太轻、位置不准、装饰线缺失 |
| 图表形态 | 柱子高低关系错误、折线点位不准、饼图角度和透视不对 |
| 配色关系 | 主色偏差、浅色底不够浅、深蓝不够深、灰度不一致 |
| 文案编排 | 段落太密、行宽过长、字号偏小、字重错误 |
| 细节元素 | 虚线、边框、图例、编号、图标、底栏、角标缺失或错位 |

人工复核后，必须把观察结论写下来，再进入下一轮修改。不要依赖记忆，也不要只说“感觉还差一点”，而应明确写成“右侧栏底色应改为纯深蓝”“标题需要上移 8px”“第二组柱图数据顺序与原图相反”这类可执行结论。

## 8. 记录规范

每次分析与修改，都应在 `records/` 目录下留下简明但明确的记录。记录的重点不是流水账，而是帮助下一轮继续接手。建议至少写明当前页的原图特点、当前截图的主要偏差、下一轮应优先处理的部分。

推荐记录结构如下。

| 记录项 | 内容 |
| :--- | :--- |
| 页面编号 | 例如 `slide_131` |
| 原图概述 | 说明这页是何种版式、核心视觉元素是什么 |
| 当前偏差 | 写出最明显的三到五个问题 |
| 已完成修改 | 本轮改了哪些结构、颜色、图表或排版 |
| 下一步计划 | 还需要继续调整哪些地方 |

## 9. 素材生成与 assets 管理规范

高端品牌类模板中大量使用了产品摄影、人物生活方式照片、背景纹理、装饰图形等视觉素材。这些素材**不能用纯色背景或 SVG 占位符代替**，必须通过 AI 图像生成工具产出与原图风格匹配的真实素材，才能达到视觉还原的目标。

### 9.1 素材生成原则

| 原则 | 说明 |
| :--- | :--- |
| 按需生成 | 每张幻灯片在原图分析阶段即识别出所需素材，分析完成后立即生成 |
| 风格匹配 | 生成提示词应还原原图的摄影风格、光线、色调和主体特征，不得随意替换为不相关图片 |
| 多素材支持 | 一张幻灯片可能需要多张独立素材（如背景图、产品图、人物图、装饰纹理），每张单独生成 |
| 不允许 Base64 内嵌大图 | 素材图片必须以独立文件存储，HTML 通过相对路径引用，禁止将大图 Base64 内嵌到 HTML 中 |

### 9.2 assets 目录结构

每个模板的素材统一存放在该模板目录下的 `assets/` 子目录，按幻灯片编号组织：

```
02高端品牌PPT参考/
  slides/                          # 原始参考图片（只读，不修改）
    01_perfume_luxury_brand/
      slide_01.jpg
      ...
  html/                            # 复刻 HTML 文件
    01_perfume_luxury_brand/
      slide_01.html
      ...
  assets/                          # AI 生成素材
    01_perfume_luxury_brand/
      slide_01/                    # 该页所需的所有素材
        perfume_bottle.png
        background_texture.jpg
        ...
      slide_02/
        ...
  screenshots/                     # 截图输出
    01_perfume_luxury_brand/
      slide_01.png
      ...
```

### 9.3 HTML 本地素材引用规范

HTML 文件中引用素材时，使用相对路径指向 `assets/` 目录，路径格式如下：

```html
<!-- HTML 位于 html/01_perfume_luxury_brand/slide_01.html -->
<!-- 素材位于 assets/01_perfume_luxury_brand/slide_01/perfume_bottle.png -->
<img src="../../assets/01_perfume_luxury_brand/slide_01/perfume_bottle.png" ...>
```

CSS `background-image` 同理：

```css
background-image: url('../../assets/01_perfume_luxury_brand/slide_01/background_texture.jpg');
```

所有素材路径均使用**相对路径**，不使用 CDN 或绝对路径，确保本地截图和 GitHub 仓库中均可正常渲染。

## 10. GitHub 提交规范

本项目所有产出文件（HTML、assets 素材、screenshots 截图）均需提交到 GitHub 仓库，以便持续追踪进度和版本对比。

### 10.1 提交内容

| 目录 | 是否提交 | 说明 |
| :--- | :--- | :--- |
| `html/` | ✅ 必须提交 | 所有复刻 HTML 文件 |
| `assets/` | ✅ 必须提交 | 所有 AI 生成的素材图片 |
| `screenshots/` | ✅ 必须提交 | 每张幻灯片的截图，用于视觉对比存档 |
| `slides/` | ✅ 已存在，不修改 | 原始参考图片，只读 |
| `records/` | ✅ 必须提交 | 分析记录与进度文档 |

### 10.2 提交频率与粒度

- 每完成一个模板（如 `01_perfume_luxury_brand` 全部55张）后，执行一次 `git add + commit + push`
- commit message 格式：`feat: 完成 {模板名} slide_{起始}~slide_{结束} HTML 复刻`
- 例如：`feat: 完成 01_perfume_luxury_brand slide_01~slide_55 HTML 复刻`

### 10.3 .gitignore 注意事项

确认 `assets/` 和 `screenshots/` 目录下的图片文件不在 `.gitignore` 排除范围内，若存在排除规则需手动添加例外。

## 11. 依赖与环境说明

本项目的截图流程依赖 Playwright 与 Chromium。若环境缺失，可执行如下命令安装：

```bash
sudo pip3 install playwright pillow
python3.11 -m playwright install chromium
```

这些依赖仅用于**渲染页面和生成截图**，不用于做任何图像相似度分析。

## 12. 最终执行口径

后续在本项目中执行切片优化任务时，统一采用以下口径：

> **允许使用工具完成截图；禁止使用脚本进行图像对比；所有优化判断只能建立在人工查看原图与截图的基础上。**

如果后续有人继续维护本项目，必须严格遵守这一工作流。若发现有人重新引入机器评分、相似度对比或自动验收逻辑，应视为偏离项目标准，需要回退到人工看图流程。

通过遵循上述规范，切片还原工作才能真正围绕“视觉一致性”展开，而不是被不可靠的机器分数牵着走。
